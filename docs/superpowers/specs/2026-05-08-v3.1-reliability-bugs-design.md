# v3.1 Reliability Bugs — Design

Date: 2026-05-08
Branch: `fix/v3.1-reliability-bugs`
Target version: `3.1.0` (minor, additive)

## Summary

Single PR fixing 4 reliability/throughput bugs identified during performance audit:

1. **TCS leak** in `RabbitMqRpcTransport.OnResponseReceivedAsync` — caller hangs to timeout when reply deserialization fails.
2. **Broken graceful shutdown** in `RpcServerHostedService.StopAsync` (no-op) and `RabbitMqRpcTransportHost.Dispose` (empty).
3. **`RpcRequest.RequestId` regenerated** on server-side deserialization (alloc Guid+string per request, then overwritten).
4. **`ConsumerDispatchConcurrency` not configured** — defaults to 1 in RabbitMQ.Client v7. Server processes messages sequentially per channel even with `PrefetchCount > 1`.

## Scope

- **In scope:** 4 fixes + 4 tests in one PR.
- **Out of scope:** publish channel pool, LINQ removal in dispatcher, `StableTypeName` cache, double-serialization (`JsonElement`) — separate follow-ups.

## Detailed Fixes

### Fix #1 — TCS leak on poison reply

`RabbitMqRpcTransport.OnResponseReceivedAsync`: extract `correlationId` and `TryRemove` from `pending` BEFORE deserialize. On any deserialization error, the TCS receives the exception. Caller fails fast instead of waiting for timeout.

```csharp
private Task OnResponseReceivedAsync(object sender, BasicDeliverEventArgs ea)
{
    var correlationId = ea.BasicProperties.CorrelationId;
    if (string.IsNullOrWhiteSpace(correlationId))
        return Task.CompletedTask;

    if (!pending.TryRemove(correlationId, out var tcs))
        return Task.CompletedTask;

    try
    {
        var response = serializer.Deserialize<RpcResponse>(ea.Body);
        if (response is null)
            tcs.TrySetException(new RpcProtocolException("Empty RPC response."));
        else
            tcs.TrySetResult(response);
    }
    catch (Exception ex)
    {
        tcs.TrySetException(ex);
        logger.LogError(ex, "Error deserializing RPC response. CorrelationId: {Id}", correlationId);
    }
    return Task.CompletedTask;
}
```

Race note: if timeout already removed the TCS, `TryRemove` returns `false` and the late reply is dropped silently. Correct.

`RpcProtocolException` — new exception type in `AsbtCore.Broker.Core.Exceptions`.

### Fix #2 — Graceful shutdown

**Interface change** (`IRpcTransportHost`):

```csharp
public interface IRpcTransportHost : IAsyncDisposable
{
    Task StartAsync(
        Func<RpcRequest, CancellationToken, Task<RpcResponse>> handler,
        IReadOnlyCollection<string> routes,
        CancellationToken ct = default);

    Task StopAsync(CancellationToken ct = default) => Task.CompletedTask;  // default impl, non-breaking
}
```

**`RabbitMqRpcTransportHost` state additions:**

- `List<(IChannel channel, string consumerTag)> consumers` (replaces `List<IChannel> channels`).
- `int inFlightCount` (Interlocked).
- `TaskCompletionSource drained` (lazy, set when shutdown starts and inFlight > 0).
- `volatile bool stopping`.

**`StartAsync`:** capture consumer tag returned from `BasicConsumeAsync`, store with channel.

**`HandleIncomingAsync`:** wrap body — `Interlocked.Increment(ref inFlightCount)` at start. In `finally`: `if (Interlocked.Decrement(ref inFlightCount) == 0 && stopping) drained.TrySetResult();`.

**`StopAsync`:**

1. `stopping = true`.
2. For each `(channel, consumerTag)`: `await channel.BasicCancelAsync(consumerTag, ct)` — broker stops dispatching.
3. If `inFlightCount == 0` → return.
4. Else `await drained.Task.WaitAsync(ct)` — bounded by caller's `ct`.
5. `await DisposeAsync()`.

**`Dispose()` (sync):** `DisposeAsync().AsTask().GetAwaiter().GetResult()` (last resort path; preferred — `DisposeAsync`).

**`RpcServerHostedService.StopAsync`:** `await transportHost.StopAsync(cancellationToken);` (replaces no-op).

### Fix #3 — `RequestId` not regenerated on deserialize

- `RpcContracts.cs`: `public string RequestId { get; set; } = default!;`
- `RpcClient.BuildRequest`: explicit assignment:
  ```csharp
  var request = new RpcRequest
  {
      RequestId = Guid.NewGuid().ToString("N"),
      InterfaceName = interfaceName,
      MethodName = method.Name
  };
  ```

### Fix #4 — `ConsumerDispatchConcurrency` configured

`RpcOptions.cs`:

```csharp
public ushort? ConsumerDispatchConcurrency { get; set; }
```

`RabbitMqRpcTransportHost.StartAsync`:

```csharp
var dispatch = options.ConsumerDispatchConcurrency ?? options.PrefetchCount;
var channelOptions = new CreateChannelOptions(
    publisherConfirmationsEnabled: true,
    publisherConfirmationTrackingEnabled: true,
    consumerDispatchConcurrency: dispatch);
```

`null` default → falls back to `PrefetchCount`. Existing deployments without config change get parallel dispatch up to current prefetch.

## Tests

Location: `Tests/AsbtCore.Broker.ClientServer.Tests`. Use existing real-RabbitMQ test harness; if absent, gate with `[Trait("Category", "Integration")]`.

### Test #1 — `Transport_PoisonReply_FailsCallerImmediately`

- Start transport, declare reply queue.
- `SendAsync` with valid route; server publishes back invalid JSON to `replyTo`.
- Assert: caller throws (`JsonException` / `RpcProtocolException`) **before** `DefaultTimeoutSeconds`.
- Assert: `pending` empty after the call (probe via subsequent `SendAsync` with same `RequestId` succeeds — would fail with "Duplicate request id" if leak persisted).

### Test #2 — `Server_Shutdown_DrainsInFlightAndCancelsConsumer`

- Server with slow handler (`await Task.Delay(2_000, ct)`).
- Client sends 3 parallel requests.
- After 500ms: `await host.StopAsync(ct)`.
- Assert: all 3 client tasks complete with success **before** `StopAsync` returns (drain).
- Assert: `SendAsync` after `StopAsync` returns timeout (no consumer on the route).

### Test #3 — `RequestId_NotRegenerated_OnDeserialize`

- Round-trip: serialize `RpcRequest { RequestId = "abc", … }` to JSON via `RpcJson.Options`, deserialize back. Assert `RequestId == "abc"`.
- Deserialize JSON without `requestId` field → `RequestId == null` (default!).
- `RpcClient.BuildRequest` (via reflection or test helper): two consecutive calls produce different `RequestId` values (not empty, not equal).

### Test #4 — `ConsumerDispatchConcurrency_AppliedToChannel`

- `RpcOptions.ConsumerDispatchConcurrency = 8`, `PrefetchCount = 8`.
- Server handler: `bag.Add(Environment.CurrentManagedThreadId); await Task.Delay(200);` returns OK.
- Client: 32 parallel `SendAsync`.
- Assert: `bag.Distinct().Count() > 1` — parallel dispatch verified.
- Without fix (baseline run with `ConsumerDispatchConcurrency = 1`): assert distinct count == 1.

## Risks

- **Parallel handlers in user code.** New default behavior dispatches concurrently up to `PrefetchCount`. User implementations must be thread-safe. Mitigation: explicit migration note in README; opt-out via `ConsumerDispatchConcurrency = 1`.
- **Drain hang on stuck handler.** `StopAsync` waits on `drained` task bounded by caller's `ct`. External shutdown deadline (e.g., host's `ShutdownTimeout`) decides.
- **`Dispose()` sync over async.** Last-resort path. Documented; users should prefer `DisposeAsync`.
- **Race in `TryRemove` (Fix #1).** Late reply after timeout — drops silently. Acceptable: caller already failed, no double-resolution.

## Migration (README addendum, v3.0 → v3.1)

> ### v3.1
>
> - **Server-side message dispatch is now parallel by default.** `ConsumerDispatchConcurrency` defaults to `PrefetchCount`. Existing handlers must be thread-safe. Set `RpcOptions.ConsumerDispatchConcurrency = 1` to retain v3.0 sequential behavior.
> - `IRpcTransportHost.StopAsync(CancellationToken)` added (default interface method — implementations not required to override, but should for graceful drain).
> - `IRpcTransportHost.Dispose()` is now a sync-over-async last-resort path. Prefer `DisposeAsync()`.
